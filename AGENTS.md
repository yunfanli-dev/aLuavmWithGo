# AGENTS.md

本文件约束后续 agent 在本仓库中的工作方式。

## 启动规则

每次开启新的 session，必须先阅读以下文件：

1. [README.md](./README.md)
2. [ProgressLog.md](./ProgressLog.md)
3. [Milestones.md](./Milestones.md)

目标：

- 先理解项目简略说明
- 明确当前项目做到哪一步了
- 明确当前项目处在哪个里程碑
- 从当前进度继续，不重复做已经完成的内容

## 阅读升级规则

如果仅通过 `README.md` 仍无法看明白项目目标、范围、架构或下一步方向，必须继续阅读：

4. [ProjectPlan.md](./ProjectPlan.md)

原则：

- 先读简略文档
- 不清楚再读完整项目书
- 不要在没读清楚上下文前直接改代码

## 执行规则

- 默认从 `ProgressLog.md` 记录的“当前 / 下一步”继续推进
- 每次只推进一个清晰的小阶段，避免一次改动过散
- 新增或修改项目中的方法时，必须补充中文简体注释，至少说明方法用途、预期行为；如果实现不直观，也要补充关键实现说明
- 新增结构体、类型、关键字段、重要 helper 或关键执行路径时，也必须补充中文简体注释，不要只给方法写注释
- 仓库中的代码注释默认统一使用中文简体；后续新增注释、修改已有注释或补充 `TODO` 时，都不得再写英文注释
- 扫描到未完成、占位、临时实现的代码时，必须在对应位置添加中文简体 `TODO` 与简洁说明，明确后续要补什么
- 完成一部分后，必须更新 [ProgressLog.md](./ProgressLog.md)
- 每次改动项目后，必须检查并按需更新 [Milestones.md](./Milestones.md)
- 每次改动项目后，必须检查并按需更新 [Lua51SyntaxSupport.md](./Lua51SyntaxSupport.md)
- 每次完成阶段性工作后，必须明确标注当前属于哪个里程碑；不能只写“当前 / 下一步”而不写里程碑
- 如果里程碑推进、完成或切换，必须同步更新 [Milestones.md](./Milestones.md) 中对应阶段的状态说明
- 每次新增、修改或确认一种可重复执行的测试方法后，必须检查并按需更新 [test/README.md](./test/README.md)
- 更新进度时保持极简，只写：时间、完成、当前、当前里程碑、下一步、备注

## 进度更新规则

每次完成阶段性工作后，必须同步更新进度，包括：

- 本次完成了什么
- 当前处于什么阶段
- 当前属于哪个里程碑
- 接下来准备做什么
- 是否有阻塞或验证限制

要求：

- 尽量简短
- 不写长篇分析
- 不重复抄项目书
- `ProgressLog.md` 的每一条记录必须显式包含“当前里程碑”
- 如果判断项目已从一个里程碑进入下一个里程碑，必须同时更新 `Milestones.md`

## 工作优先级

1. 先看 `README.md`
2. 再看 `ProgressLog.md`
3. 再看 `Milestones.md`
4. 看不明白再看 `ProjectPlan.md`
5. 按当前进度继续
6. 完成阶段性工作后更新 `ProgressLog.md`
7. 为新增或修改的方法、类型、关键结构补充中文简体注释，并尽量写清用途、预期行为和关键实现点
8. 为未完成或占位代码补中文简体 `TODO` 与简洁说明
9. 每次改动后检查 `Milestones.md` 是否需要更新
10. 每次改动后检查 `Lua51SyntaxSupport.md` 是否需要更新
11. 每次更新进度时显式写明当前里程碑
12. 每次涉及测试流程、验证命令、排障步骤变化时检查 `test/README.md` 是否需要更新

## 禁止事项

- 不读取进度就直接开始大改
- 不读取里程碑就直接开始大改
- 忽略 `ProgressLog.md` 里的当前状态
- 新增或修改方法却不补注释
- 新增、修改或保留英文代码注释，而不是统一改为中文简体
- 发现未完成代码却不标 `TODO`
- 做完工作不检查 `Milestones.md` 是否需要更新
- 做完工作不检查 `Lua51SyntaxSupport.md` 是否需要更新
- 做完工作不标注当前里程碑
- 做完测试相关改动不检查 `test/README.md` 是否需要更新
- 做完工作不更新进度
- 在 `README.md` 已足够清楚时无意义重复通读所有文档
