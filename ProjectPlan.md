# ProjectPlan

## 1. 项目定位

`aLuavmWithGo` 是一个使用 Go 实现的 Lua 虚拟机项目。

核心目标不是做一个完整的 Lua 发行版，而是实现一套可嵌入 Go 工程的 Lua 运行时，使宿主程序可以：

- 加载 Lua 代码
- 执行 Lua 脚本
- 在 Go 与 Lua 之间传递数据
- 逐步扩展宿主能力与脚本能力

项目最终希望成为一个适合嵌入业务系统的轻量 Lua 执行层，而不是单纯的实验性解析器。

## 2. 总体目标

项目整体分为三个递进目标：

### 2.1 最小可运行目标

- 可以从 Go 中加载一段 Lua 代码
- 可以完成基础语句执行
- 可以输出执行结果或错误信息

### 2.2 可嵌入目标

- Go 宿主可以创建 Lua VM 实例
- Go 可以向 Lua 注入基础函数
- Lua 可以调用部分宿主暴露能力

### 2.3 可扩展目标

- 运行时结构清晰，便于继续补齐语言特性
- 支持测试、调试、错误定位
- 为后续标准库扩展和宿主桥接保留清晰边界

## 3. 技术路线

首版建议采用“自己实现前端 + 自己实现解释执行虚拟机”的路线，而不是依赖外部 Lua 解释器。

推荐路线：

1. Lua 源码输入
2. 词法分析
3. 语法分析
4. 生成内部中间表示
5. 交给 VM 执行

这样做的原因：

- 可控性更强，适合学习和逐步补齐能力
- 能清晰掌握 Lua 执行模型
- 更方便未来做裁剪、宿主扩展和调试输出

不建议首版同时兼顾“源码解释执行”和“官方 Lua 字节码兼容”两条路线。首版应先聚焦源码加载执行，等核心运行时稳定后，再评估是否支持 Lua chunk。

## 4. 技术框架

建议项目采用分层结构：

### 4.1 接口层

负责对外暴露嵌入能力，例如：

- 创建虚拟机实例
- 加载脚本字符串或文件
- 执行脚本
- 注册宿主函数

这一层要尽量稳定，后续内部实现变化不应频繁影响外部调用方式。

### 4.2 前端层

负责 Lua 源码处理：

- Lexer：词法分析
- Parser：语法分析
- AST 或中间表示构建

这部分的职责是把 Lua 代码转成内部可执行结构，而不是直接处理运行时逻辑。

### 4.3 编译或转换层

这一层负责把 AST 转成更适合 VM 的表示，例如：

- 指令序列
- 常量表
- 函数原型
- 局部变量与作用域信息

如果首版不做完整字节码，也建议做一层中间表示，避免 parser 直接驱动执行逻辑，减少后期耦合。

### 4.4 虚拟机层

这是项目核心，负责：

- Value 表示
- 操作数栈
- 调用帧
- 程序计数器
- 指令分发
- 函数调用与返回

这一层决定项目是否真正具备“可运行的 Lua VM”能力。

### 4.5 运行时层

负责支撑语义行为：

- 全局环境
- table 基础行为
- 闭包和 upvalue
- 内建函数
- 错误处理

运行时层与 VM 层边界要尽量清楚，避免所有逻辑都堆进指令执行器。

### 4.6 测试层

测试至少分为三类：

- 词法和语法单元测试
- VM 指令执行测试
- 端到端脚本执行测试

项目越早建立测试样例，后续扩展越稳。

## 5. 推荐目录规划

首版可以采用如下结构：

```text
.
├── cmd/
│   └── aluavm/
├── internal/
│   ├── api/
│   ├── lexer/
│   ├── parser/
│   ├── ir/
│   ├── vm/
│   ├── runtime/
│   └── stdlib/
├── examples/
├── test/
├── README.md
├── ProgressLog.md
├── Milestones.md
└── ProjectPlan.md
```

说明：

- `cmd/aluavm` 用于放最小运行入口，便于手动验证。
- `internal/api` 对外整合加载和执行流程。
- `internal/ir` 放内部可执行表示。
- `internal/vm` 聚焦执行机理。
- `internal/runtime` 聚焦语言语义与环境。
- `examples` 放脚本样例。
- `test` 放测试说明与脚本。

## 6. 首版范围建议

首版不追求完整 Lua 语义，建议先完成一个可运行子集。

优先支持：

- `nil`、`boolean`、`number`、`string`
- 局部变量
- 基础算术运算
- 比较与条件跳转
- `if`、`while`
- 简单函数定义与调用
- `return`
- 最小 `print`

延后支持：

- 完整 table 语义
- metamethod
- coroutine
- module 体系
- 完整标准库
- 官方 Lua 字节码兼容

这种切分可以降低首版复杂度，优先打通主链路。

## 7. 开发难点

这个项目的难点主要不在“把代码跑起来”，而在“让结构足够清晰，后面能继续扩展”。

### 7.1 语义与执行模型映射

Lua 语言表面简单，但执行模型并不简单。局部变量作用域、函数调用、返回值、多值语义、闭包捕获等都需要在内部结构上提前考虑。

### 7.2 Value 设计

Lua 是动态类型语言，Go 是静态类型语言。如何用 Go 表达统一的 Lua 值模型，是整个 VM 的基础。这里设计不稳，后续 table、函数、闭包都会变得难维护。

### 7.3 栈帧与调用约定

函数调用、参数传递、返回值处理、局部变量布局，需要一套稳定的调用约定。这个部分很容易在后续扩展时反复推翻，所以首版就要定清楚。

### 7.4 table 与闭包

table 是 Lua 最核心的数据结构，闭包是 Lua 运行时的重要特性。这两部分实现难度较高，建议在主链路跑通后再逐步补齐。

### 7.5 错误处理与调试

如果没有清晰的错误位置信息、调用栈信息、脚本上下文，后面调试会非常痛苦。即使首版功能有限，也应预留错误定位能力。

## 8. 关键设计原则

- 先打通链路，再补语言特性
- 先保证结构清晰，再追求完整兼容
- parser、IR、VM、runtime 分层明确
- 所有新增核心方法都写简洁注释
- 所有暂未完成的占位实现都加 `TODO`
- 每推进一个小阶段就更新进度和验证方式

## 9. 阶段规划

### M1 基础骨架

- 初始化 `go.mod`
- 建立基础目录结构
- 提供最小 CLI 或 demo 入口
- 定义 `State`、`Value`、`Stack` 基础对象

交付标准：

- 项目可以编译
- 可以创建 VM 实例
- 可以跑一个最小演示入口

### M2 加载链路

- 实现 lexer
- 实现 parser
- 生成 AST 或 IR
- 打通从源码到内部表示的转换

交付标准：

- 能读取并解析简单 Lua 脚本
- 能输出结构化的内部表示

### M3 执行核心

- 实现基础指令或节点执行
- 完成栈、帧、PC 管理
- 支持基础表达式与流程控制

交付标准：

- 能执行简单算术、变量、条件、循环脚本

### M4 运行时扩展

- 支持函数调用
- 支持基础全局环境
- 提供最小宿主函数注册机制
- 逐步补充 table 或闭包能力

交付标准：

- Go 可以注入函数
- Lua 可以完成基本函数调用

### M5 验证与整理

- 补齐测试
- 增加样例脚本
- 整理错误处理
- 补齐开发文档

交付标准：

- 主链路具备稳定验证方式
- 项目对后续迭代者足够清晰

## 10. 验证策略

建议从一开始就建立可重复验证方式：

- 用 `go test` 跑单元测试
- 用样例脚本跑集成测试
- 用 `cmd/aluavm` 做手工验证

建议最早准备的测试样例：

- 常量求值
- 变量赋值
- 算术表达式
- `if` 分支
- `while` 循环
- 函数调用与返回
- Go 注入 `print` 或自定义函数

## 11. 风险控制

项目推进时要特别避免以下问题：

- 一开始追求完整 Lua 兼容，导致范围失控
- parser 和 VM 强耦合，导致后续难扩展
- 没有测试样例，改动后无法确认回归
- table、闭包、标准库同时推进，导致复杂度爆炸
- 没有宿主接口边界，后续嵌入方式混乱

## 12. 总结

这个项目的总体策略应该是：

- 先做一个结构清晰、能真正执行 Lua 子集的 Go VM
- 再逐步扩展语言能力和宿主集成能力
- 始终围绕“可嵌入 Go 项目”这个目标推进

只要主链路是清楚的，即“Lua 源码输入 -> 内部表示 -> VM 执行 -> Go 宿主交互”，后续功能扩展都会更稳。
